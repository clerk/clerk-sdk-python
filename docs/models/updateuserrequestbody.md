                                                                                                               | The hashing algorithm that was used to generate the password digest.<br/>The algorithms we support at the moment are [bcrypt](https://en.wikipedia.org/wiki/Bcrypt), [bcrypt_sha256_django](https://docs.djangoproject.com/en/4.0/topics/auth/passwords/),<br/>[md5](https://en.wikipedia.org/wiki/MD5), pbkdf2_sha256, pbkdf2_sha512, [pbkdf2_sha256_django](https://docs.djangoproject.com/en/4.0/topics/auth/passwords/),<br/>[phpass](https://www.openwall.com/phpass/), [scrypt_firebase](https://firebaseopensource.com/projects/firebase/scrypt/),<br/>[sha256](https://en.wikipedia.org/wiki/SHA-2), [scrypt_werkzeug](https://werkzeug.palletsprojects.com/en/3.0.x/utils/#werkzeug.security.generate_password_hash)<br/>and the [argon2](https://argon2.online/) variants argon2i and argon2id.<br/><br/>If you need support for any particular hashing algorithm, [please let us know](https://clerk.com/support).<br/><br/>Note: for password hashers considered insecure (at this moment MD5 and SHA256), the corresponding user password hashes will be transparently migrated to Bcrypt (a secure hasher) upon the user's first successful password sign in.<br/>Insecure schemes are marked with `(insecure)` in the list below.<br/><br/>Each of the supported hashers expects the incoming digest to be in a particular format. Specifically:<br/><br/>**bcrypt:** The digest should be of the following form:<br/><br/>`$<algorithm version>$<cost>$<salt & hash>`<br/><br/>**bcrypt_sha256_django:** This is the Django-specific variant of Bcrypt, using SHA256 hashing function. The format should be as follows (as exported from Django):<br/><br/>`bcrypt_sha256$$<algorithm version>$<cost>$<salt & hash>`<br/><br/>**md5** (insecure): The digest should follow the regular form e.g.:<br/><br/>`5f4dcc3b5aa765d61d8327deb882cf99`<br/><br/>**pbkdf2_sha256:** This is the PBKDF2 algorithm using the SHA256 hashing function. The format should be as follows:<br/><br/>`pbkdf2_sha256$<iterations>$<salt>$<hash>`<br/><br/>Note: Both the salt and the hash are expected to be base64-encoded.<br/><br/>**pbkdf2_sha512:** This is the PBKDF2 algorithm using the SHA512 hashing function. The format should be as follows:<br/><br/>`pbkdf2_sha512$<iterations>$<salt>$<hash>`<br/><br/>  _iterations:_ The number of iterations used. Must be an integer less than 420000.<br/>  _salt:_ The salt used when generating the hash. Must be less than bytes.<br/>  _hash:_ The hex-encoded hash. Must have been generated with a key length less than 1024 bytes.<br/><br/>**pbkdf2_sha256_django:** This is the Django-specific variant of PBKDF2 and the digest should have the following format (as exported from Django):<br/><br/>`pbkdf2_sha256$<iterations>$<salt>$<hash>`<br/><br/>Note: The salt is expected to be un-encoded, the hash is expected base64-encoded.<br/><br/>**pbkdf2_sha1:** This is similar to pkbdf2_sha256_django, but with two differences:<br/>1. uses sha1 instead of sha256<br/>2. accepts the hash as a hex-encoded string<br/><br/>The format is the following:<br/><br/>`pbkdf2_sha1$<iterations>$<salt>$<hash-as-hex-string>`<br/><br/>**phpass:** Portable public domain password hashing framework for use in PHP applications. Digests hashed with phpass have the following sections:<br/><br/>The format is the following:<br/><br/>`$P$<rounds><salt><encoded-checksum>`<br/><br/>- $P$ is the prefix used to identify phpass hashes.<br/>- rounds is a single character encoding a 6-bit integer representing the number of rounds used.<br/>- salt is eight characters drawn from [./0-9A-Za-z], providing a 48-bit salt.<br/>- checksum is 22 characters drawn from the same set, encoding the 128-bit checksum with MD5.<br/><br/>**scrypt_firebase:** The Firebase-specific variant of scrypt.<br/>The value is expected to have 6 segments separated by the $ character and include the following information:<br/><br/>_hash:_ The actual Base64 hash. This can be retrieved when exporting the user from Firebase.<br/>_salt:_ The salt used to generate the above hash. Again, this is given when exporting the user.<br/>_signer key:_ The base64 encoded signer key.<br/>_salt separator:_ The base64 encoded salt separator.<br/>_rounds:_ The number of rounds the algorithm needs to run.<br/>_memory cost:_ The cost of the algorithm run<br/><br/>The first 2 (hash and salt) are per user and can be retrieved when exporting the user from Firebase.<br/>The other 4 values (signer key, salt separator, rounds and memory cost) are project-wide settings and can be retrieved from the project's password hash parameters.<br/><br/>Once you have all these, you can combine it in the following format and send this as the digest in order for Clerk to accept it:<br/><br/>`<hash>$<salt>$<signer key>$<salt separator>$<rounds>$<memory cost>`<br/><br/>**scrypt_werkzeug:** The Werkzeug-specific variant of scrypt.<br/><br/>The value is expected to have 3 segments separated by the $ character and include the following information:<br/><br/>_algorithm args:_ The algorithm used to generate the hash.<br/>_salt:_ The salt used to generate the above hash.<br/>_hash:_ The actual Base64 hash.<br/><br/>The algorithm args are the parameters used to generate the hash and are included in the digest.<br/><br/>**argon2i:** Algorithms in the argon2 family generate digests that encode the following information:<br/><br/>_version (v):_ The argon version, version 19 is assumed<br/>_memory (m):_ The memory used by the algorithm (in kibibytes)<br/>_iterations (t):_ The number of iterations to perform<br/>_parallelism (p):_ The number of threads to use<br/><br/>Parts are demarcated by the `$` character, with the first part identifying the algorithm variant.<br/>The middle part is a comma-separated list of the encoding options (memory, iterations, parallelism).<br/>The final part is the actual digest.<br/><br/>`$argon2i$v=19$m=4096,t=3,p=1$4t6CL3P7YiHBtwESXawI8Hm20zJj4cs7/4/G3c187e0$m7RQFczcKr5bIR0IIxbpO2P0tyrLjf3eUW3M3QSwnLc`<br/><br/>**argon2id:** See the previous algorithm for an explanation of the formatting.<br/><br/>For the argon2id case, the value of the algorithm in the first part of the digest is `argon2id`:<br/><br/>`$argon2id$v=19$m=64,t=4,p=8$Z2liZXJyaXNo$iGXEpMBTDYQ8G/71tF0qGjxRHEmR3gpGULcE93zUJVU`<br/><br/>**sha256** (insecure): The digest should be a 64-length hex string,
